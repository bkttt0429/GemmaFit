### 更新後的競爭對手解讀

直接的動作/教練競爭對手不再只有 FormForward 和 Maestro。將 **PhysioGemma** 加入為直接的相鄰競爭對手，因為它佔據了「運動處方 / 物理治療」的語境。這也是一個有用的警告：評審將會看到運動 + 健康專案，所以 GemmaFit 的文案必須明確區分：

- **GemmaFit 測量什麼：** 姿態推導的動作證據、階段/事件、可見度、信心度、次數/平衡視窗。
- **GemmaFit 不聲明什麼：** 診斷、復健結果、跌倒風險評分、肌少症、疼痛治療、受力/EMG/GRF、臨床運動處方。
- **GemmaFit 輸出什麼：** 有界限的指導、棄權原因、照護日誌，以及帶有證據參考的可分享摘要。

最強的非直接競爭對手不是那些擁有最大模型的專案。而是那些具有可靠產品邊界的專案：Vestige 引述來源的私人記錄、Marunthagam 的確定性安全底線，以及 Android LLM 伺服器專案的模型管理清晰度。GemmaFit 應該複製這些產品紀律模式，同時保持獨特的攝影機姿態證據路徑。

### 立即的 2026-05-12 建議

1. 將 **PhysioGemma** 加入競爭對手投影片中，作為新的直接健康/運動相鄰競爭對手。
2. 增加一個文案段落，將 GemmaFit 與物理治療/分診應用程式進行對比：GemmaFit 是健康和動作證據支援，而不是醫療處方。
3. 強化展示 UI 中關於證據來源的部分：
   - `由姿態規則測量` (Measured by pose rules)
   - `Layer 2 事件` (Layer 2 event)
   - `由本地 Gemma 生成` (Generated by local Gemma)
   - `模板後備` (Template fallback)
   - `棄權：信心度低` (Abstained: low confidence)
4. 借鑑 Android LLM 伺服器專案的模型就緒用語：
   - 模型找到/缺失 (model found/missing)
   - 載入中/預熱中 (loading/warmup)
   - 後端名稱 (backend name)
   - 後備原因 (fallback reason)
   - 延遲 / 事件計數 (latency / event count)
5. 在展示或文案中加入一個小的「聲明邊界」卡片。醫療/物理治療競爭對手的增加，使得 GemmaFit 更安全的邊界變得更加重要，而非無關緊要。

## 2026-05-12 深度競爭對手功能分析

本節不看儲存庫數量，而是專注於強大的競爭對手能實際展示什麼：輸入證據、模型部署、控制層、證明產物 (proof artifacts)，以及 GemmaFit 應該複製或避免的功能模式。

### 分析軸向

| 軸向 | 檢查重點 | 為何對 GemmaFit 很重要 |
|---|---|---|
| 領域聲明 (Domain claim) | 健康、教育、分診、復健、安全稽核、照護者支援 | 高影響力的領域很擁擠。GemmaFit 需要一個清晰的非診斷性動作支援聲明。 |
| 輸入證據 (Input evidence) | 姿態地標 (Pose landmarks)、照片、文字問診、生命徵象、螢幕狀態、上傳的媒體、語音 | 擁有觀察證據的專案比純聊天應用程式更強大。GemmaFit 最強的證據是即時姿態推導的動作。 |
| 模型部署 (Model deployment) | Android 上的 LiteRT-LM、Ollama 桌面端、託管 API、瀏覽器模型、伺服器路由 | 評審會獎勵真實裝置的證明。GemmaFit 應該強調 Pixel/本地執行並披露後備狀態。 |
| 控制層 (Control layer) | 確定性規則、協議引擎、函式呼叫、JSON schema、拒絕策略 | 強大的安全專案不會讓模型自由運作。GemmaFit 應該展示 C++ 規則、Layer 2 事件和 FC 驗證作為控制層。 |
| 證明產物 (Proof artifacts) | 基準測試 (Benchmarks)、保留評估 (held-out evals)、螢幕截圖、APK、延遲、校準、測試 | SiteSafe、SilverAid 和 Marunthagam 很危險，因為它們展示了證明，而不僅僅是展示概念。 |
| 弱點 (Weakness) | 聲明風險、桌面端依賴、無即時路徑、無評估、範圍過大 | GemmaFit 應該攻擊狹窄的缺口：即時長者動作、證據收據和安全的聲明邊界。 |

### GemmaFit 仍能獲勝的地方

1. **即時長者動作證據。** 大多數健康競爭對手是文字/照片分診或照護助手。GemmaFit 可以展示即時的姿態推導事件：起立坐下完成、平衡保持開始/完成、低信心度棄權。
2. **僅限 Pixel 的有界限管線。** SilverAid 和 Maestro 使用較大的主機模型進行總結。如果核心迴圈在 Pixel 上離線工作並帶有事件驅動的 E2B，GemmaFit 可以更簡單且更符合競賽要求。
3. **非診斷性安全邊界。** PhysioGemma/OfflineMedic/Marunthagam 更接近臨床領域。GemmaFit 應該將克制轉化為信任特徵：健康教練和可分享證據，沒有跌倒風險/復健/肌少症聲明。
4. **函式呼叫加上證據參考。** SiteSafe 是這裡的標竿。GemmaFit 需要相當的證明，證明 Gemma 輸出與動作證據視窗綁定的受限工具/模板。
5. **照護者/專業人員分享報告。** 這是一個不需要臨床聲明的高影響力展示面。報告可以說明觀察到了什麼、跳過了什麼，以及使用者主觀報告了什麼。

### 這次競爭對手審查後的主要風險

| 風險 | 為何重要 | 截止日期前的緩解措施 |
|---|---|---|
| 無真實裝置證明 | 即便領域深度較淺，Android 執行階段競爭對手看起來也可能更可靠。 | 記錄 Pixel 冒煙測試 (smoke test)：模型找到、後端、一個事件觸發的響應、一個棄權案例。 |

| 臨床用語漂移 | 物理治療/醫療競爭對手讓評審對不安全聲明敏感。 | 在文案/展示中放置可見的「產品聲明矩陣」，並在提示詞/測試中強制執行被阻擋的標籤。 |

## 大多數團隊在做什麼

主流模式是「針對脆弱環境的離線/本地優先助手」：健康、鄉村醫療、教育、危機支援、災難應對、詐騙防護或無障礙設施。許多團隊將 Gemma 定位為專門工作流程之上的自然語言層，但公開的詮釋資料通常沒有顯示硬性的驗證閘門、真實裝置部署或確定性安全邊界。

有很多健康專案，但大多數是醫療問診、分診、症狀、藥物或公共衛生情報應用程式。這對影響力框架來說是好事，但從聲明和安全角度來看風險也較高。GemmaFit 應該避免偏向診斷，並將產品聲明保持在非診斷性的動作品質和照護日誌支援。

教育領域很擁擠。通用的離線家教不太可能脫穎而出，除非它有強大的部署證明、個人化或新穎的介面。GemmaFit 的長者雙重任務模式 (dual-task mode) 與教育僅有輕微重疊；更強大的故事是與健康相鄰的保健 (wellness) 加上保護隱私的裝置端證據。

直接的動作/教練競爭對手很稀少。最接近 GemmaFit 的可見專案是 FormForward 和 Maestro。FormForward 是關於跑步姿勢分析；Maestro 是給兒童的 AI 小提琴教練。兩者似乎都不是具備確定性生物力學和證據契約的離線 Pixel 長者動作品質系統。

## 本次掃描後的 GemmaFit 定位

最強的路徑不是「通用 AI 健身教練」。許多儲存庫已經聲明本地優先、弱勢群體、健康和教育。GemmaFit 應該定位為：

> 一個保護隱私、在裝置端執行的長者動作支援系統，將攝影機姿態證據和自我報告的吃力程度轉換為有界限的指導、照護日誌，以及供照護者/專業人員分享的摘要，而不做醫療聲明。

獲勝條件應該是具體的展示證據：

- 真實的 Android/Pixel 路徑，而不僅僅是桌面端 Python 或 Web 展示。
- 在模型輸出之前的確定性證據閘門。
- Layer 2 時序階段/事件證據，而非單一影格模板判斷。
- 具備拒絕和證據參考檢查的函式呼叫/路由驗證。
- 長者英雄流程：起立坐下、平衡/步態觸碰、雙重任務提示、主觀簽到、人物角色報告。
- 清晰的產品聲明矩陣：沒有跌倒風險評分、診斷、受力、EMG、GRF、心率狀態、肌少症或復健進度。

### 給 GemmaFit 的 P0 建議

1. 在每個 LLM 可見輸出中新增一個 `EvidenceReceipt` 物件：
   - `session_id`
   - `trigger`
   - `activity_context`
   - `motion_feature_window`
   - `layer2_event`
   - `capability_contract`
   - `evidence_refs`
   - `backend`
   - `fallback`
   - `unsupported_claims_blocked`

2. 在除錯和照護者報告中新增可見的後端/來源標籤：
   - `由姿態規則測量` (Measured by pose rules)
   - `由本地 Gemma 生成` (Generated by local Gemma)
   - `模板後備` (Template fallback)
   - `棄權：信心度低` (Abstained: low confidence)

3. 在增加更多模型訓練之前，先增加本地模板/RAG 資產：
   - `senior_activity_templates.json`
   - `unsupported_claims_matrix.json`
   - `caregiver_report_templates.json`
   - `dual_task_prompt_bank.json`

4. 增加離線匯出路徑：
   - 照護者文字摘要
   - 專業人員分享的結構化摘要
   - 用於除錯的原始證據 JSON

5. 增加模型就緒畫面或除錯卡片：
   - `.litertlm` 找到/缺失
   - 後端載入狀態
   - 上次冒煙測試結果
   - 後備原因

### 這次截止日期前應避免的模式

- 來自 OfflineMedic/VillageDoc 類型專案的完整醫療分診用語。這會增加風險，且不符合 GemmaFit 的證據邊界。
- Always-on YOLO 或 Always-on E2B 影片推理。這會損害延遲/電池表現，而且如果 Layer 2 事件採樣有效，這是不必要的。
- 作為 P0 功能的雙裝置大型模型路由。SilverAid 和 Maestro 有效地使用了它，但 GemmaFit 應首先證明僅限 Pixel 的路徑。
- 原始的「模型看影片並決定」架構。GemmaFit 應該將姿態推導證據和能力閘門置於所有生成內容的前面。

## 對 GemmaFit 的立即影響

1. 不要將其當作通用醫療助手來競爭。這個領域很擁擠且有聲明風險。
2. 不要將其當作通用家教來競爭。教育類有最多的複製品和膚淺的變體。
3. 讓展示看起來明顯是在真實裝置上運行且基於證據的。許多儲存庫聲稱「離線」或「裝置端」；很少有公開描述能證明 Pixel 等級的部署和確定性閘門。
4. 投入長者照護的動作證據。SilverAid 和失智症/照護助手在受眾上很接近，但他們似乎並未掌握來自攝影機姿態的動作品質證據。
5. 將 FormForward 留在競爭對手投影片中。這是最清晰的姿勢分析重疊點；GemmaFit 的回應應該是時序證據、長者安全邊界和 Android 離線執行。
6. 將安全邊界作為一項功能。市場上充滿了有過度聲明醫療分診或預測風險的專案。GemmaFit 可以透過拒絕不支援的聲明和展示稽核軌跡，看起來更成熟可靠。
